# 표준 통합 저울 데이터 수집 프로그램 — 개발 가이드

> OJT 과제 학습 정리 문서
> 목표: 전자저울이 USB로 보내는 무게 데이터를 PC가 받아서 → 숫자만 추출 → 화면 표시 + 클립보드 자동 복사

---

## 전체 그림

```
저울 ─(USB-Serial)─> PC → [내 프로그램] → 무게 추출 → 클립보드 → ERP 붙여넣기
```

핵심 흐름: **"데이터를 어떻게 받느냐(시리얼)" → "어떻게 숫자만 뽑느냐(파싱)" → "어떻게 보여주고 복사하느냐(GUI+클립보드)"**

## 먼저 알아야 할 핵심 용어

### 시리얼 통신 관련

저울은 데이터를 **시리얼 통신(Serial Communication)** 으로 보냅니다. 한 번에 1비트씩 순서대로 전기 신호를 보내는 방식인데, 양쪽(저울과 PC)이 **똑같은 규칙(통신 파라미터)** 으로 맞춰져 있어야 글자가 안 깨집니다.

| 용어 | 뜻 | 이 과제에서 |
|------|-----|------------|
| **Baud rate (보드레이트)** | 초당 신호 전송 속도 | 과제에서 `9600` bps 사용 |
| **Data bits (데이터 비트)** | 글자 하나를 몇 비트로 표현하나 | 보통 8 |
| **Parity (패리티)** | 오류 검출용 추가 비트 | 보통 None |
| **Stop bit (정지 비트)** | 글자 끝을 알리는 비트 | 보통 1 |
| **COM Port** | Windows가 USB 시리얼 장치에 붙여주는 이름 | `COM3` 같은 식. 장치관리자에서 확인 |
| **USB-Serial** | USB를 옛날 시리얼(RS-232)처럼 쓰게 해주는 변환 | 저울 안/케이블에 변환칩이 들어있음 |

> **중요:** 한 COM 포트는 **한 프로그램만** 열 수 있습니다. 아두이노 IDE의 시리얼 모니터가 포트를 잡고 있으면 내 프로그램이 그 포트를 못 엽니다. 테스트할 때 꼭 닫으세요.

### 데이터 형식

CAS 저울(CK200SC)은 이런 한 줄 문자열을 반복해서 보냅니다:

```
ST,GS,   37.39,kg
```

- `ST` = Stable(무게가 안정됨), 흔들리는 중이면 `US`(Unstable)
- `GS` = Gross(총중량), `NT`면 Net(순중량)
- `37.39` = **우리가 뽑아야 할 무게 숫자**
- `kg` = 단위

여기서 `37.39`만 정확히 추출하는 게 **파싱(Parsing)** 단계입니다.

## 개발 순서 (단계별)

추천 전략: **"하드웨어 없이 먼저 다 만들고, 마지막에 진짜 신호로 검증"**

### 0단계 — 환경 준비
- Python 설치 (3.10+ 권장)
- 라이브러리 설치: `pip install pyserial`
- 프로젝트 폴더 구조 잡기

### 1단계 — 시리얼로 데이터 받아보기 (가장 먼저)

`pyserial` 라이브러리로 COM 포트를 열고, 저울이 보내는 줄을 그대로 콘솔에 찍어봅니다.

```python
import serial
ser = serial.Serial('COM3', 9600, timeout=1)
line = ser.readline().decode('ascii').strip()  # "ST,GS,   37.39,kg"
print(line)
```

- **`readline()`** = 줄 단위로 읽기
- **`decode()`** = 바이트(전기신호 원본)를 사람이 읽는 글자로 변환
- "포트 열기 → 한 줄 받기"만 되면 통신의 절반은 끝.

### 2단계 — 시뮬레이터로 테스트 환경 만들기 (하드웨어 없이)

진짜 저울이 없으니 가짜 데이터를 만들어 1단계를 테스트.

- **방법 A — 파이썬 시뮬레이터**: 또 다른 파이썬 스크립트가 `ST,GS,   37.39,kg` 문자열을 계속 시리얼로 내보냄.
- **가상 시리얼 포트(Virtual COM Port)**: PC 안에서 COM3↔COM4를 가상으로 연결(예: `com0com`). 한쪽으로 시뮬레이터가 쓰고, 다른 쪽으로 내 프로그램이 읽음.

> **가상 포트** = 실제 하드웨어 없이 소프트웨어로 만든 가짜 COM 포트 한 쌍.

### 3단계 — 무게 숫자 추출 (파싱 / 정규식)

받은 줄에서 `37.39`만 뽑습니다. **정규식(`re` 모듈)** 사용.

```python
import re
m = re.search(r'(-?\d+\.?\d*)', line)   # 숫자 부분 찾기
weight = m.group(1)  # "37.39"
```

- **정규식** = "이런 패턴의 문자열을 찾아라" 규칙. `\d`=숫자, `\.`=점, `*`=0개 이상.
- 안정(`ST`)일 때만 복사하게 할지 같은 규칙도 여기서 처리.

### 4단계 — GUI 만들기 (Tkinter)

`Tkinter` 는 Python 기본 내장 GUI 라이브러리(추가 설치 X). 무게를 크게 보여주는 창, 포트 선택, 시작/정지 버튼 등.

- **위젯(Widget)** = 버튼/라벨/입력칸 같은 화면 부품
- **이벤트 루프(`mainloop`)** = 창이 계속 떠 있게 유지하는 반복문

### 5단계 — 멀티스레드 (화면이 안 멈추게)

`readline`은 데이터가 올 때까지 **멈춰서 기다립니다(blocking)**. GUI랑 같은 곳에서 돌리면 창이 얼어붙음. 그래서:

- **메인 스레드** = 화면(Tkinter)
- **백그라운드 스레드** = 시리얼 수신 (`threading` 모듈)

용어:
- **스레드(Thread)** = 한 프로그램 안에서 동시에 도는 작업 흐름
- **블로킹(Blocking)** = 작업이 끝날 때까지 다음으로 안 넘어가고 기다리는 것
- 두 스레드가 데이터를 주고받을 땐 **Queue(큐)** 를 쓰면 안전.

### 6단계 — 클립보드 자동 복사 (이 과제의 핵심 기능)

추출한 무게를 클립보드에 넣으면, 작업자가 ERP/메모장에서 `Ctrl+V`로 붙여넣음.

- 방법 1: Tkinter 내장 (`clipboard_clear()` / `clipboard_append()`)
- 방법 2: `pyperclip` 라이브러리 (`pip install pyperclip`, 더 간단)

### 7단계 — 설정 파일 / 로깅 / 예외 처리

- **설정 파일(JSON)**: 포트/보드레이트 등을 코드에 박지 말고 `config.json`에 저장 → 다음 실행 때 읽어옴 (`json` 모듈)
- **로깅(Logging)**: `print` 대신 `logging` 모듈로 동작 기록을 파일에 남김 → 현장 문제 추적용
- **예외 처리(Exception Handling)**: `try/except` 로 "포트 없음", "케이블 빠짐" 상황에서 프로그램이 죽지 않게

### 8단계 — 아두이노로 실물 신호 검증 (하드웨어 실습)

진짜 시리얼 신호로도 잘 되는지 확인. Arduino UNO에 저울 흉내 코드를 올림.

```cpp
void setup() { Serial.begin(9600); }
void loop() {
  Serial.println("ST,GS,   37.39,kg");
  delay(1000);
}
```

- **Arduino IDE** = 아두이노에 코드를 작성/업로드하는 프로그램
- **스케치(Sketch)** = 아두이노 코드
- 업로드 후 **시리얼 모니터 꼭 닫기** (안 그러면 내 프로그램이 포트 못 엶)

### 9단계 — exe로 배포 (PyInstaller)

현장 PC엔 Python이 안 깔려 있어도 돌아가야 하므로 `.exe` 하나로 묶음.

```
pip install pyinstaller
pyinstaller --onefile --windowed main.py
```

- `--onefile` = 파일 하나로
- `--windowed` = 콘솔창 안 뜨게(GUI 앱용)

# 확장 (2일차) — 식품 배합 칭량 POP

> **1일차(위 0~9단계):** 저울 1대 → 무게 받아 화면 표시 + 클립보드 복사
> **2일차(이 장):** 저울 여러 대 → 레시피(배합비)대로 **목표±허용오차를 판정**하며
>                    원료를 순서대로 다는 **식품 공정용 계량(POP) 단말**

## 왜 확장했나

식품 공장 **배합 공정**에서는 무게를 단순히 "보는" 게 아니라, 레시피대로 원료를
**정량(목표±허용오차)** 으로 투입하고 그 실적을 **배치(LOT) 단위로 기록**한다.
배합비가 곧 품질·안전과 직결되고, 리콜 대비 **추적성**이 필수이기 때문이다.

→ 그래서 1일차의 "무게 표시기"를 **"배합 칭량 가이드(POP)"** 로 확장했다.

## 확장의 핵심 원리 — 코어는 그대로, 위에 계층을 얹는다

1일차에서 만든 **수신·파싱 파이프라인은 한 줄도 바꾸지 않고 재사용**한다.
그 위에 "판정 → 레시피 → POP 화면" 계층만 새로 얹었다.

```
[1일차 코어 — 그대로 재사용]
 저울 ─> serial_reader(수신) ─> scale_parser(파싱) ─> WeightData(무게, 안정)
                                                          │
[2일차 확장 — 위에 얹음]                                    ▼
 recipe(배합비: 목표·허용·포트) ──> judge(목표±허용 판정) ──> pop_gui(POP 화면)

 (기존 main.py 무게표시기는 그대로 두고, pop_main.py 라는 새 진입점을 병렬로 추가)
```

→ **"기존 부품을 안 부수고 새 부품을 조립"** 한 것이 이번 확장의 방식이다.
   덕분에 무게표시기 모드(`main.py`)와 배합칭량 모드(`pop_main.py`)가 공존한다.

| 파일 | 역할 | 1일차 대비 |

## 확장 단계 (어떻게 이루어졌나)

### E1 — 가상 저울 2대 환경 (com0com)
하드웨어 없이 저울 2대를 흉내내려고 **가상 시리얼 포트 쌍 2개**(COM5↔COM6, COM7↔COM8)를 만들었다.
- Windows 11 보안(메모리 무결성·Secure Boot)이 구형 드라이버를 막아 **코드 52** 발생
  → **Pete Batard 재서명 드라이버**로 해결.
- 배운 점: 커널 드라이버 서명/보안 정책, 회사 PC 설정 변경 시 **승인 절차**의 중요성.

### E2 — 파서 업그레이드 (`scale_parser`)
- **체크섬(NMEA XOR)** 검증으로 전송 중 변조된 데이터를 거부
- `ST/US` 로 시작하지 않는 **쓰레기 데이터 거부** (아무 숫자나 무게로 받지 않음)
- 단위를 **숫자 뒤에서** 찾아 `GS` 의 `G` 를 단위 `g` 로 오인식하는 것 방지
- 체크섬 없는 신호(아두이노)도 **하위호환**으로 그대로 허용

### E3 — 다중 제조사 형식 (`scale_formats`)
- 받는 쪽(`parse_weight`)은 이미 범용이라 여러 형식을 처리한다. 부족한 건 **보내는 쪽**.
- 그래서 제조사별 **송신 형식**을 한 곳에 모아 시뮬레이터가 골라 보내게 했다.
  - CAS: `ST,GS,   37.39,kg*HH` (체크섬) / A&D: `ST,+00037.39 kg` (부호+0채움, 체크섬 없음)

### E4 — 칭량 시나리오 시뮬레이터
- 무게를 무작위로 흔들던 것 → **0 → 부음(증가, US) → 목표 근처 안정(ST) → 비움** 으로 변경
- 실제 칭량처럼 "차오르다 멈춤"을 재현. 가끔 **과충전**을 섞어 '초과' 판정 데모.

### E5 — 판정 로직 (`judge`)
- `순중량 = 계량값 − 용기중량`, `차이 = 순중량 − 목표`, `오차% = 차이/목표 × 100`
- 판정: **부족**(차이<−허용) / **양호**(|차이|≤허용) / **초과**(차이>+허용)
- **통과 가능 = 안정(ST) AND 양호** ← '통과' 버튼이 열리는 게이트
- 화면과 분리된 **순수 함수**라 하드웨어/GUI 없이 단위 테스트로 검증.

### E6 — 배합비 (`recipe`)
- `recipe.json` 에 제품·배치LOT·원료목록(목표·허용·저울포트·제조사) 정의.
- 실무에선 DB(MES/ERP)에 있지만 학습용으론 JSON. **`load_recipe()` 한 곳만** 바꾸면
  나중에 DB(SQLite/SQL)로 교체 가능하게 데이터 접근을 분리.

### E7 — POP 화면 (`pop_gui` / `pop_main`)
- 식품 MES '계량 입력' 화면을 본떠: 자재명·목표중량·제품로트·자재로트 +
  큰 **계량값** + **판정 바(신호등 색)** + **통과 / 취소 / 무시저장** 버튼.
- 부품 조립: `serial_reader`(수신) + `scale_parser`(파싱) + `judge`(판정) + `recipe`(배합비).

### E8 — 원료 순차 + 배치/LOT 전환
- 원료1(저울1, COM6) 통과 → **포트를 바꿔** 원료2(저울2, COM8)로 자동 진행.
- 모든 원료 끝 → **배치 완료** → **[다음 배치 시작]** 으로 LOT 번호를 1 증가(`P-…-01 → -02`)
  새 배치를 처음 원료부터 반복. (1 배치 = 1 생산 LOT)

## 실행 (배합 칭량 POP)

```
# 저울1 — 원료1 (CAS, 목표 10kg) → 앱은 COM6 수신
python simulator/simulator.py COM5 CAS 10
# 저울2 — 원료2 (A&D, 목표 0.7kg) → 앱은 COM8 수신
python simulator/simulator.py COM7 AND 0.7
# POP 화면
python src/pop/pop_main.py
```

> 소스는 역할별 패키지로 정리됨 — **common**(공유: paths·app_logger·scale_parser·serial_reader)
> / **pop**(배합 칭량) / **display**(무게 표시기). 시뮬레이터·scale_formats 는 `simulator/` 로 분리.

## 데이터 흐름 한눈에

```
저울1(COM5→COM6) ─┐
                  ├─ serial_reader ─ scale_parser ─ WeightData ─┐
저울2(COM7→COM8) ─┘  (원료 순서대로 한 번에 한 포트)              │
                                                               ▼
                            recipe(목표·허용) ──> judge ──> pop_gui
                                                               │
                              부족/양호/초과 판정 + 통과 → 다음 원료 → 배치(LOT) 완료
```
